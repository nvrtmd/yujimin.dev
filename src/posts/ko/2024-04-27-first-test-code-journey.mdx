## 1. 테스트 코드와의 조우

"테스트 코드를 왜 써야 할까?"라는 질문에서 내 이야기는 시작된다. 프로젝트의 QA 이후, 팀은 한동안 새로운 기능 개발보다는 리팩토링이나 코드 변경을 최소화하는 작업에 집중하고 있었다. 어느 날, 팀 회의에서 앞으로 각자가 주도적으로 맡을 역할을 정하던 중, 나는 TDD 담당자를 자청했다.

당시에 우리 팀이 작업하는 코드베이스에는 테스트 코드가 거의 없었다. 주요 함수 몇 개에만 유닛 테스트가 작성되어 있을 뿐, 전체적으로 테스트 커버리지는 매우 낮았다. 그래서 내가 직접 코드를 수정할 때마다 혹시라도 기존과 다르게 동작해서 예상치 못한 버그가 생기면 어쩌나 하는 불안감이 항상 마음 한구석에 있었다. 테스트 코드가 있으면 리팩토링이나 유지보수가 훨씬 편하지 않을까? 이런 생각도 들었다. 그때까지만 해도 테스트 코드 작성 경험이 전무한 상태였지만, 코드 수정 시 기존 동작과 달라져서 생기는 버그를 방지하고, 리팩토링이나 유지보수에 도움이 될 거란 확신이 있었다. 그래서 이번 기회에 내가 직접 테스트 코드 작성을 주도해보기로 마음먹었다.

이 글은 나처럼 테스트 코드에 대한 경험이 거의 없어서 막막한 주니어 프론트엔드 개발자들에게, 내가 겪은 시행착오와 작은 성공, 그리고 얻은 교훈을 공유하고자 쓴 글이다.

## 2. 무엇을 테스트할 것인가

테스트 코드를 작성하기로 결심한 뒤, 가장 먼저 한 일은 코드베이스를 샅샅이 살펴보는 것이었다. 우리 팀은 통계 데이터를 받아 차트로 보여주는 대시보드 프로덕트를 만들고 있다. 코드를 훑어보던 중, 나는 프로덕트의 핵심 로직 중 하나인 `summarizeData` 함수를 발견했다. 이 함수는 하루 동안 수집된 여러 데이터 배열을 하나의 객체로 요약해주는 역할을 하는 함수이다. `summarizeData`는 input과 output이 명확하게 타입 인터페이스로 정의되어 있으며, 외부 모듈이나 함수에 의존하지 않아 같은 input을 넣으면 항상 같은 output이 나오는 순수 함수였다. 무엇보다도, 이 함수는 우리 서비스의 핵심 동작을 책임지고 있었기 때문에, 여기에 테스트 코드를 추가하면 팀 전체에 큰 도움이 될 것 같았다. 이 정도면 테스트 코드 작성의 첫 타깃으로 딱이라는 생각이 들었다.

## 3. 첫 시도: 막막함, 그리고 작은 성공

하지만, 막상 `summarizeData` 함수의 140줄이 넘는 방대한 코드에 압도당했다. 입사 후 지금까지는 이 함수로부터 나온 데이터를 받아서 사용하는 부분만 다뤄왔지, 내부 로직을 이렇게 자세히 들여다본 건 처음이었다. input과 output 객체의 구조도 복잡했다. 객체의 속성이 여러 개이고, 경우에 따라 존재하거나 존재하지 않는 속성도 있었으며, 특정 속성들은 또 다른 객체를 value로 가지고 있었다. 즉, 내부 로직에 분기가 많고, 다소 난해한 구조였다. 또한 함수 내부에는 다양한 분기와 조건문이 얽혀 있었다.

테스트를 작성하려면 무엇보다 함수에 대한 철저한 이해가 필요했다. 그래서 함수의 모든 줄을 읽으며, "이 부분은 어떤 역할을 하는지", "이 조건문은 어떤 상황에서 동작하는지" 등을 주석으로 작성하기 시작했다. 그렇게 코드를 해부하듯 분석하다 보니, 함수의 전체 흐름과 내부 분기, 각 속성의 의미를 조금씩 파악할 수 있었다.

이제 테스트 코드를 작성할 차례였다. 하지만 테스트 코드를 한 번도 제대로 작성해본 적이 없던 나는, 어디서부터 시작해야 할지 막막했다. 그래서 ChatGPT에게 "테스트를 처음 작성하는 사람을 위한 단계별 가이드"를 요청했다. ChatGPT는 "가장 간단한 조건, 테스트하기 쉬운 케이스부터 시작해보라"고 조언했다.

그래서 나는 함수 내부에서 input 객체의 특정 속성이 없을 때, output 객체의 해당 속성이 NaN이 되어야 한다는 조건을 골랐다. 실제로 input 객체에서 한 속성을 제거한 뒤, `summarizeData`에 넣고, output의 해당 속성이 NaN이 되는지 `expect(...).toBeNaN()`으로 검증했다. 테스트를 실행해보니, 무사히 통과했다! 처음으로 테스트 코드를 작성해보고, 첫 성공을 경험한 순간이었다. 이 작은 성공이 나에게 큰 동기부여가 되었고, 조금씩 감이 잡히기 시작했다.

## 4. 난관: 복잡한 input과 테스트 설계의 어려움

하지만, 테스트 케이스를 하나 만들고 나니 금세 또 다른 벽에 부딪혔다. 함수가 워낙 크고 분기가 많아, 어떤 테스트 케이스가 필요한지 코드와 주석만으로는 쉽게 알기 어려웠다. 게다가 input 객체의 속성도 많고, 그 중 일부는 객체를 값으로 가지기도 하여서 매 테스트 케이스마다 input 객체를 새로 만드는 것도 번거롭고 가독성이 떨어질 것 같았다.

## 5. 체계적 접근: 테스트 케이스 도출 및 효율화 전략

이대로는 안 되겠다 싶어서, 함수의 모든 분기와 조건을 표로 정리해보기로 했다. 각 row는 하나의 테스트 케이스가 되고, column에는 테스트 레벨(최상위 속성), 테스트 제목, 연관 조건, 기대 결과, 테스트 여부를 적었다.

예를 들면 다음과 같다.

| 테스트 레벨 | 테스트 제목 | 연관 조건 | 기대 결과 | 테스트 여부 |
|---|---|---|---|---|
| all fields | input에 `score.normal` 속성이 없을 경우(undefined) `all.normal`은 `NaN`이어야 한다 | `input.score.normal === undefined` | `summary.all.normal === NaN` | O |
| | input에 `result`가 "Normal"일 때, `all.result.normal`의 값을 1 증가시켜야 한다 | `input.result === 'Normal'` | `summary.all.result.normal++` | X |
| | input에 `result`가 "Abnormal"일 때, `all.result.abnormal`의 값을 1 증가시켜야 한다 | `input.result === 'Abnormal'` | `summary.all.result.abnormal++` | X |

이렇게 표로 정리하니, 어떤 케이스를 테스트해야 하고, 어떤 케이스가 아직 남았는지 한눈에 볼 수 있었다. 테스트 케이스의 누락도 방지할 수 있었다.

또한, 반복되는 input 준비의 비효율을 줄이기 위해 `baseInput`이라는 객체를 만들었다. 모든 테스트 케이스에서 공통으로 사용하는 속성만 미리 채워넣고, 각 테스트에서는 필요한 속성만 `baseInput`을 복사해서 커스텀했다.

예를 들어, 아래와 같이 `baseInput`을 전역적으로 선언한 뒤,

```typescript
const baseInput: InputType = {
  id: 'abc',
  timestamp: 1673017200,
  processing_time: {
    // ...
  },
  score: {
    // ...
  },
  // ...
};
```

테스트 케이스에서는 전개 구문을 사용하여 `baseInput`과 테스트 조건에 해당하는 객체 속성을 조합하여 테스트용 input 객체를 만들어서 테스트하는 방법이다.

```typescript
test('should set all.normal to NaN when score.normal is missing (undefined)', () => {
  const input: InputType = {
    ...baseInput,
    score: {
      a: 9.7,
      b: 99.59,
      c: 4.2,
    },
  };

  const testCase = summarizeData(input);

  expect(testCase.all.normal).toBeNaN();
});
```

이렇게 하니 테스트 코드가 훨씬 간결해지고, 반복 작업도 줄어 편리하였다.

## 6. 실행: 테스트 코드 작성과 커버리지 100% 달성

이제 표에 정리한 테스트 케이스를 하나씩 구현하며, 체크박스로 완료 여부를 관리했다. 이 방식으로 약 15개에 이르는 모든 테스트 케이스를 빠르게 작성할 수 있었다.

테스트 작성을 완료한 뒤, 테스트를 실행하여 모든 테스트 케이스가 잘 통과되는 것을 확인하였다. 그리고 마지막으로 테스트 커버리지 리포트를 생성해보니, Statements, Branches, Functions, Lines 모든 항목에서 100%를 달성했다. 내가 직접 복잡한 로직을 분석하고, 테스트를 설계하고, 체계적으로 모든 분기를 테스트하여 커버리지 100%를 달성하니 내가 작성한 테스트 코드가 코드베이스의 안정성을 높였다는 확신이 들어 정말 뿌듯하고 기분이 좋았다.

## 7. 회고: 얻은 교훈과 주니어 개발자에게 전하는 팁

물론 커버리지 100%가 모든 버그를 잡아준다는 의미는 아니다. 그렇지만 앞으로 이 함수를 수정하거나 리팩토링할 때, 내가 작성한 테스트 코드가 최소한의 방어선이 되어줄 것이라는 믿음이 생겼다. 또한, 나중에 새로운 팀원이 합류하여 코드를 파악할 때도 내가 작성한 테스트 코드가 함수의 동작을 이해하는 데 작은 도움을 보탤 수 있을 것 같다. 테스트 코드를 통해 함수의 동작과 분기, 기대 결과를 명확하게 파악할 수 있기 때문이다.

나처럼 테스트 코드 작성 경험이 없는 주니어 개발자라면, 복잡한 로직에 대한 테스트 코드를 작성할 때 다음과 같은 방식을 추천한다.

1. 테스트 대상 함수를 철저히 분석하기
2. 가능한 모든 분기를 표로 정리해서 테스트 케이스를 도출하기
3. 반복되는 입력값은 base 객체로 효율화하기

이렇게 하면 막막함을 줄이고, 체계적으로 테스트 코드를 작성할 수 있을 것이다.

## 8. 마무리: 성장의 기록

이번 경험을 통해 테스트 코드가 단순히 "커버리지"를 채우는 도구가 아니라, 개발자로서의 성장과 코드의 신뢰성을 높여주는 중요한 자산임을 깨달았다. 처음엔 많이 막막했었지만, 막상 당장 필요한 일이 무엇인지를 생각해내고 작은 작업부터 단계 별로 해내자 생각보다 할 만 했다. 이렇게 직접 부딪혀보며 얻은 자신감은 앞으로의 커리어에 큰 자산이 될 것이다. 앞으로도 팀의 TDD 담당자로서, 적극적으로 테스트 코드 도입과 개선에 힘쓸 계획이다.
