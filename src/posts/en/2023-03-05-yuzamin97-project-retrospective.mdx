## Key Features

- **Authentication (Sign up, Login/Logout)**
  - Implemented a custom hook using `useReducer` to validate ID, password, and nickname inputs.

  - Login form inputs are controlled via a custom hook.

- **Posting**
  - Logged-in users can create posts.
  - Users can edit or delete only their own posts.
- **Comments**
  - Users can comment on posts.
  - Users can edit or delete only their own comments.
- **Post History**
  - Logged-in users can view a collected list of posts they have written.
- **User Profile**
  - Users can view and update their member information.
- **Account Deletion**
  - Users can withdraw their membership via 'My Page'.

  - Implemented via a custom hook utilizing the **IntersectionObserver API**.

## 1:N Relationships with Foreign Keys

In the previous project, the `Member` and `Post` tables were not related, making it impossible to query "posts written by a specific user" efficiently. This time, I added a `Comment` table and established 1:N relationships.

```javascript
// models/index.js
const Sequelize = require('sequelize');
// ... imports

const db = {
  sequelize,
  Sequelize,
  Post: Post(sequelize, Sequelize),
  Member: Member(sequelize, Sequelize),
  Comment: Comment(sequelize, Sequelize),
};

/*
  A member can write multiple posts -> hasMany
  Foreign key set to member_idx
*/
db.Member.hasMany(db.Post, {
  foreignKey: 'member_idx',
});

/*
  Post table depends on Member table
  Alias for the relationship set to post_writer
*/
db.Post.belongsTo(db.Member, {
  foreignKey: 'member_idx',
  as: 'post_writer',
});

/*
  Member - Comment / Post - Comment Relationships
  - One member can write multiple comments
  - One post can have multiple comments
*/
db.Member.hasMany(db.Comment, {
  foreignKey: 'member_idx',
});
db.Comment.belongsTo(db.Member, {
  foreignKey: 'member_idx',
  as: 'comment_writer',
});

db.Post.hasMany(db.Comment, {
  foreignKey: 'post_idx',
  onDelete: 'cascade',
});
db.Comment.belongsTo(db.Post, {
  foreignKey: 'post_idx',
  as: 'commented_post',
});

module.exports = db;

```

It seems simple now, but it was a significant hurdle to overcome compared to last year.

## Offset and Limit for Infinite Scroll

To implement infinite scroll, I modified the backend to respond with segmented data based on `start` (offset) and `count` (limit) passed via query strings.

```javascript
router.get('/list', async (req, res) => {
  const { start, count } = req.query;
  try {
    const postList = await Post.findAll({
      order: [['createdAt', 'DESC']],
      include: [
        // ... include models
      ],
      offset: Number(start), // From the 'start'th post
      limit: Number(count), // Fetch 'count' number of posts
    });
    return res.status(StatusCodes.OK).json({
      data: postList,
    });
  } catch {
     // ... error handling
  }
});

```

To handle cases where the client doesn't send `start` or `count` (requesting the full list), I refined the code to use `null` instead of `undefined` to explicitly represent the absence of a value.

```javascript
offset: start ? Number(start) : null,
limit: count ? Number(count) : null,

```

I encountered countless errors. This time, I made a habit of noting them down. Here are a few selected cases.

## (deprecated) event.target vs event.currentTarget

### The Problem

```typescript
const handleClick = (e: React.MouseEvent<HTMLElement>) => {
  console.log(e.target.dataset.id) // error!
  console.log(e.currentTarget.dataset.id) // board-icon
}
<div data-id="board-icon" onClick={(e) => handleClick(e)}>
  <div>A</div>
  <div>B</div>
</div>

```

I wanted to read the `data-id` of the parent `div` regardless of whether the user clicked child A or B. However, `e.target.dataset.id` threw an error because the property didn't exist on the children.

### The Solution

Changing it to `e.currentTarget.dataset.id` solved it.
`e.target` refers to the element that was actually clicked (which could be the children without the dataset), while `e.currentTarget` refers to the element the event listener is attached to (the parent `div`).

## (deprecated) Applying Styles to Active Components

### The Problem

I initially tried to manage the active state of navigation icons using separate boolean states for each icon (`boardClicked`, `documentClicked`, etc.). This was inefficient; as icons increased, so did the state variables.

### The Solution

I refactored this to use a single string state `clickedItem`.

```typescript
export const SideNavBar = () => {
  const [clickedItem, setClickedItem] = useState('')

  const handleClick = (item) => {
    setClickedItem(item)
  }

  return (
    <Wrapper>
      <NavItem
        name="Board"
        // ...
        isClicked={clickedItem === 'Board'}
        onClick={() => handleClick('Board')}
      />
      {/* ... other items */}
    </Wrapper>
  )
}

```

This ensures only the item matching the state receives the active style.

## Splitting the Button Component

### The Problem

I needed two behaviors: a standard button and a "Push Lock" button (stays pressed until toggled).

### The Solution

I split them into `PushLockButton` and `Button`.
For the standard `Button`, I implemented a UX detail where the action triggers on `mouseup`. If the user presses down but drags the mouse *out* before releasing, the action cancels (mimicking standard OS button behavior).

```typescript
export const Button = memo(
  ({ restoreHandler, name, children, type, isDisabled }: ButtonProps) => {
    // ... state logic

    const handleButtonRestore = () => {
      if (restoreHandler) {
        restoreHandler()
      }
      setIsPushed(false)
    }

    const handleButtonMouseOut = () => {
      setIsPushed(false) // Visual restore only, no action
    }

    return (
      <ButtonWrapper
        // ...
        onMouseUp={handleButtonRestore}
        onMouseOut={handleButtonMouseOut}
      >
        {children}
        {name}
      </ButtonWrapper>
    )
  }
)

```

## Custom Hook: useIntersectionObserver

```typescript
import { useState, useEffect } from 'react';

// Hook that returns isVisible (true/false) based on element intersection
export const useIntersectionObserver = (
  intersectRef: React.RefObject<HTMLDivElement>,
  optionsObject: {
    root: null;
    rootMargin: string;
    threshold: number;
  },
) => {
  const { root = null, rootMargin = '0px', threshold } = optionsObject;
  const [isVisible, setIsVisible] = useState<boolean>(false);

  const handleIntersect: IntersectionObserverCallback = (entries) => {
    // Assign the IntersectionObserverEntry object to the target variable
    const target = entries[0];

    // Update isVisible state based on intersection
    if (target.isIntersecting) {
      setIsVisible(true);
    } else {
      setIsVisible(false);
    }
  };

  useEffect(() => {
    const observer = new IntersectionObserver(handleIntersect, {
      root, rootMargin, threshold,
    });

    // If intersectRef.current exists, observe it
    if (intersectRef.current) observer.observe(intersectRef.current);

    // Disconnect on unmount
    return () => observer.disconnect();
  }, [intersectRef, root, rootMargin, threshold]);

  return { isVisible };
};

```

const inputReducer = (state: State = initialState, action: Action): State => {
  switch (action.type) {
    case 'INPUT_CHANGE':

        value: action.value,

## REM Calculation based on Font Size

Understanding the `62.5%` trick:
Default browser font size is 16px. Setting `html { font-size: 62.5%; }` makes the root font size 10px (`16 * 0.625`). Thus, `1rem = 10px`.

This makes responsive typography calculations much easier:

```css
html {
  font-size: 77%; /* 1rem = 12.32px */

  @media screen and ${theme.device.desktop} {
    font-size: 73%; /* 1rem = 11.68px */
  }
  /* ... tablet, mobile adjustments */
}

```

## CSS Positioning Context

### The Problem

My `Menu` component (absolute) was positioned relative to `BottomNavBar` (relative), causing it to appear in the wrong place on mobile.

### The Solution

I removed `position: relative` from `BottomNavBar` and added a `LayoutWrapper` div with `position: relative` that wraps the entire layout. Now, the absolute positioning of the Menu is calculated relative to the full viewport wrapper, as intended.

## Including Comment Counts in Post List

### The Problem

I initially tried to fetch the full list of comments just to display the count, which is inefficient. I then tried to use `Sequelize.fn('COUNT')` but struggled to map the data correctly to the response object.

### The Solution

The final clean solution involved `subQuery: false` and correctly placing the count function in the attributes.

```javascript
router.get('/list', async (req, res) => {
  // ...
  try {
    const postList = await Post.findAll({
      // ...
      attributes: {
        include: [
          [
            Sequelize.fn('COUNT', Sequelize.col('comments.comment_idx')),
            'comments_count',
          ],
        ],
      },
      include: [
        // ... Member model
        {
          model: Comment,
          attributes: [], // We don't need the actual comment data
        },
      ],
      subQuery: false,
      group: ['post_idx'],
    });
    // ... return response
  }
   // ...
});

```

This properly returns `comments_count` in the root of each post object without messy JS mapping.
