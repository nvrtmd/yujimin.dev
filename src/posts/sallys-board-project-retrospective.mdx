# SALLY's BOARD Retrospective

![Demo](/images/posts/sallys-board-project-retrospective/demo.gif)

## Project Record

After jumping into frontend development as a non-CS major, I constantly felt the need for backend knowledge. Collaborating with backend developers made me realize how much I didn't know — and that frustration was real. When things I built didn't work as intended, sometimes it was a frontend issue. But other times, it was something happening in the communication with the backend server. In those cases, explaining the problem was difficult, and understanding the backend developer's explanation was just as hard.

I also wanted to be able to build a complete service on my own, handling both frontend and backend. Partly because backend knowledge helps you collaborate better as a frontend developer, but also out of pure curiosity — I wanted to learn about different areas of development.

Since JavaScript was my most comfortable language, I decided to learn backend development with Node.js and Express. I studied hard through courses, and SALLY's BOARD was my first project applying what I'd learned.

CRUD practice with a bulletin board seemed like the perfect starting point. It might seem simple, but since this was my first time handling everything from frontend to backend, it was harder than I expected. I ran into various issues, big and small. But once I'd implemented all the features I'd set out to build, the difficulty made the satisfaction even sweeter. That feeling of moving forward, even just a little. This small but meaningful project will definitely help me grow.

### Folder Structure

I initially used the default project structure generated by the Express CLI, but reorganized it after looking at other projects for reference.

```
// node.js
📦node-board-project
┣ 📂apis
┃ ┣ 📂admin
┃ ┣ 📂board
┃ ┣ 📂user
┣ 📂bin
┣ 📂config
┣ 📂models
┗ 📜app.js
```

- apis: Contains API endpoints organized by feature (user management, board, login/signup, etc.). Each feature folder includes model (schema definitions), router (routing definitions), middleware (middleware definitions), controller (response and error handling), and service (core business logic) files.
- config: Database connection configuration
- models: Sequelize setup

I grouped related files together by feature, then bundled those folders under `apis` so the folder structure itself tells you what each file does. File names like `userMiddleware`, `userRouter`, `userService` are explicit about their roles.

### Heroku Deployment Gotchas

I chose Heroku for server deployment. AWS seemed risky — a few weeks earlier, I'd misconfigured something and got billed over $100, which led to some desperate emailing. So I went with Heroku because it's simple to deploy and free (most important).

Heroku deployment guides are everywhere, and I referenced them too. Instead of the basic steps, I want to share the small mistakes that ate up way too much time.

1. Create a Procfile

- The Procfile specifies the command to start the server and must be at the project root. I initially forgot to create this file. The server starts with `node ./bin/www`. I was confused whether to use `nodemon` or `node` here since I'd been using nodemon during development. Looking back, it's obvious I should've used `node`, but everything was new, so I got confused.

```
web:node ./bin/www
```

2. Fix the start script in package.json

- Another silly mistake. I should've changed the `nodemon` command to `node` before deploying, but I didn't. Won't make this mistake again.

```
"scripts": {
  "start": "node ./bin/www"
},
```

- To avoid these mistakes, install nodemon as a devDependency and use a separate `devStart` script instead of the `start` script. I need to pay more attention to the scripts section in package.json.

3. Add all .env variables to settings/Config Vars

- This is crucial and easy to forget. All environment variables defined in .env must be added to Heroku's settings/Config Vars for everything to work properly.

### Cookie Not Being Sent Properly

I implemented authentication using access tokens, but ran into cookie issues — or more precisely, CORS issues — that had haunted me in previous frontend projects.

My intention was standard cookie-based authentication: when the client attempts login, the server checks the credentials against the database. If the user is registered, it generates a JWT token, puts it in a cookie, and sends it back to the client. For security, I wanted to use HttpOnly cookies so the browser couldn't access them. But I ran into the familiar CORS error message. Here's what I tried to fix it.

1. Add CORS configuration using the cors package on the server

```jsx
const cors = require('cors')
//...

app.use(
  cors({
    origin: '*', // Allow access from all domains
    credential: true, // Allow sharing of resources requiring authentication (cookies, etc.) across domains
  }),
)
```

- After this setup, when the client sent requests with `withCredentials: true`, this error appeared:

> Access to XMLHttpRequest at 'http://aaaa.com' from origin 'http://bbb.com' has been blocked by CORS policy: The value of the 'Access-Control-Allow-Origin' header in the response must not be the wildcard '*' when the request's credentials mode is 'include'. The credentials mode of requests initiated by the XMLHttpRequest is controlled by the withCredentials attribute.

- This happens because when credentials are included in the request, `Access-Control-Allow-Origin` can't be `*`. Not great for security anyway.

2. Explicitly set the origin in response headers for authenticated requests

```jsx
res.setHeader('Access-Control-Allow-origin', process.env.FRONT_URL)
res.setHeader('Access-Control-Allow-Credentials', 'true')
```

- I stored the client domain in an environment variable and allowed access and credential sharing for requests from that domain. This sent the cookie to the client, but the client couldn't send it back to the server when needed.
- For pages like the profile page where users need to prove they're logged in, the token received from the server should be sent back in a cookie. The server sent the cookie (first image), but the `/profile` request didn't include the cookie despite having `withCredentials: true` (second image).

![Cookie received in response](/images/posts/sallys-board-project-retrospective/cookie-received-response.png)

![Cookie not sent in request](/images/posts/sallys-board-project-retrospective/cookie-not-sent-request.png)

3. Add withCredentials: true not just when sending cookies but also when receiving them (login requests, etc.)

- Adding `withCredentials: true` to requests that receive cookies (not just those that send them) solved it. The cookie was properly received and sent back to the server.
- Additionally, I'd been manually setting response headers for authenticated requests:

```jsx
res.setHeader('Access-Control-Allow-origin', process.env.FRONT_URL)
res.setHeader('Access-Control-Allow-Credentials', 'true')
```

- Repeating this code in every response was tedious. I modified the cors options instead:

```jsx
app.use(
  cors({
    origin: [process.env.FRONT_URL, 'http://localhost:3000'],
    // ...
    credentials: true,
  }),
)
```

- This eliminated the need to manually set headers for each response. Requests from the deployed client domain and localhost no longer throw 'Access to XMLHttpRequest at~' errors.

### Re-setting Cookies After Updating User Info

I was puzzled why the profile page didn't load user info correctly right after updating it. Turns out I forgot that after updating user info, the server needs to reissue the cookie with the new user data.

```jsx
await User.update(modifyUserData, { where: { user_id: signedInUserId } })

const newUserData = { userId: modifyUserData.user_id }

const newToken = jwt.sign(newUserData, process.env.JWT_SECRET_KEY, {
  expiresIn: process.env.JWT_EXPIRE_TIME,
  issuer: process.env.JWT_ISSUER,
})

res.setHeader(
  'Set-Cookie',
  `token=${newToken}; Path=/; HttpOnly; SameSite=none; secure=true;`,
)
```

After updating user info, I now generate a new token with the updated data and send it in a cookie.

### Why Isn't the Cookie Stored in the Browser?

![Cookie stored in server domain](/images/posts/sallys-board-project-retrospective/cookie-stored-in-server-domain.png)

![Cookie not stored in browser](/images/posts/sallys-board-project-retrospective/cookie-not-stored-in-browser.png)

The cookie was definitely sent from the server, but it wasn't being stored in the browser. Even though HttpOnly cookies can't be manipulated client-side, I should've been able to view them in developer tools based on what I found online. After more research...

> When the server issues a Cookie, it can only specify its own domain or a parent domain as the Cookie's Domain value. If not specified, it defaults to the current host.

The cookie's domain is Heroku's domain, which is why it's set in the server's domain (first image) rather than the browser. Since I deployed the server to Heroku and the client to Netlify (for cost reasons), the domains can't match, so the cookie isn't set in the browser. This must be why in previous projects and internships, the server API address was the client address with just `.api` added. Anyway, the cookie not being stored in the browser didn't prevent me from implementing the features I wanted, so I closed this case after understanding the reason.
