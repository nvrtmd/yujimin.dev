# i18n Design: Korean & English Multilingual Support

**Date:** 2026-05-02  
**Status:** Approved

---

## Overview

Add Korean (`ko`) and English (`en`) multilingual support to yujimin.dev. Language is auto-detected from the browser's `accept-language` header and reflected in the URL via locale prefix. No manual language switcher UI.

---

## Scope

**Included:**

- All static UI text (labels, placeholders, button text, empty states, error messages)
- Page metadata (title, description, OpenGraph tags, Twitter card)
- Layout and navigation text (StartMenu, Taskbar, window titles)

**Excluded:**

- Blog post titles and content (MDX files)
- Resume content (fetched from Google Docs)

---

## Architecture

### Library

`next-intl` — App Router native, supports server and client components, type-safe message access.

### Directory Structure

```
src/
├── app/
│   ├── layout.tsx           # Root layout: defines <html lang> using getLocale(), <body>
│   ├── [locale]/
│   │   ├── layout.tsx       # Locale layout: provides NextIntlClientProvider (no <html>/<body>)
│   │   ├── page.tsx
│   │   ├── blog/
│   │   │   └── [slug]/
│   │   ├── about-me/
│   │   ├── resume/
│   │   ├── guestbook/
│   │   └── colophon/
│   ├── api/                 # Unchanged (no locale needed)
│   ├── robots.ts
│   ├── sitemap.ts
│   └── favicon.ico
├── i18n/
│   ├── routing.ts           # Single source: locales, defaultLocale, prefix strategy
│   ├── navigation.ts        # Locale-aware Link, useRouter, usePathname wrappers
│   └── request.ts           # Per-request locale resolution + message loading
└── middleware.ts            # Locale detection and routing (stays at src/middleware.ts)

messages/
├── en.json                  # Base schema (used for type inference)
└── ko.json
```

### html lang Handling

Only the root layout can render `<html>`. To set `lang` dynamically:

```ts
// src/app/layout.tsx
import { getLocale } from 'next-intl/server';

export default async function RootLayout({ children }) {
  const locale = await getLocale();
  return (
    <html lang={locale}>
      <body>{children}</body>
    </html>
  );
}
```

`src/app/[locale]/layout.tsx` does NOT re-render `<html>`. It only provides `NextIntlClientProvider` for client components:

```ts
// src/app/[locale]/layout.tsx
import { NextIntlClientProvider } from 'next-intl';
import { getMessages } from 'next-intl/server';

export default async function LocaleLayout({ children }) {
  const messages = await getMessages();
  return <NextIntlClientProvider messages={messages}>{children}</NextIntlClientProvider>;
}
```

### i18n Config Files

**`src/i18n/routing.ts`**

```ts
import { defineRouting } from 'next-intl/routing';

export const routing = defineRouting({
  locales: ['en', 'ko'],
  defaultLocale: 'en',
});
```

**`src/i18n/navigation.ts`**

```ts
import { createNavigation } from 'next-intl/navigation';
import { routing } from './routing';

export const { Link, redirect, usePathname, useRouter } =
  createNavigation(routing);
```

**`src/i18n/request.ts`**

Validates locale against the supported list before loading messages. Unsupported locales (e.g. `/fr/blog`) trigger `notFound()`, producing a controlled 404 instead of a runtime import failure.

```ts
import { getRequestConfig } from 'next-intl/server';
import { notFound } from 'next/navigation';
import { routing } from './routing';

export default getRequestConfig(async ({ requestLocale }) => {
  const locale = await requestLocale;

  if (!locale || !routing.locales.includes(locale as 'en' | 'ko')) {
    notFound();
  }

  return {
    locale,
    messages: (await import(`../../messages/${locale}.json`)).default,
  };
});
```

## Middleware

Location: `src/middleware.ts`

**Behavior:**

- All non-locale app paths → detect `accept-language` header → redirect to locale-prefixed equivalent
  - `/` → `/ko` or `/en`
  - `/blog` → `/ko/blog` or `/en/blog`
  - `/blog/some-post` → `/ko/blog/some-post` or `/en/blog/some-post`
  - `/about-me`, `/resume`, `/guestbook`, `/colophon` and all their sub-paths → same pattern
- `/ko/*`, `/en/*` → pass through
- `/api/*`, `/_next/*`, static files → skip middleware entirely

**Language detection logic:**

- `accept-language` contains `ko` → redirect to `/ko/...`
- All other languages → redirect to `/en/...` (default)

`next-intl`'s `createMiddleware(routing)` handles this automatically, including nested paths like `/blog/[slug]`.

`ko.json` has identical key structure with Korean values.

**Metadata key pattern:** Every namespace includes a `metadata` sub-key with `title` and `description`. Pages use these in `generateMetadata`:

```ts
export async function generateMetadata({ params }) {
  const { locale } = await params;
  const t = await getTranslations({ locale, namespace: 'blog.metadata' });
  return {
    title: t('title'),
    description: t('description'),
    openGraph: { title: t('title'), description: t('description') },
  };
}
```

## Type Safety

`messages/en.json` is used as the base schema. TypeScript connects message key types from this file, providing:

- Key autocomplete in IDE
- Compile-time error on typos or missing keys

Setup in `src/global.d.ts`:

```ts
import en from '../messages/en.json';

declare module 'next-intl' {
  interface AppConfig {
    Messages: typeof en;
  }
}
```

## Usage Patterns

**Server components (default):**

```ts
import { getTranslations } from 'next-intl/server';

export default async function GuestbookApp() {
  const t = await getTranslations('guestbook');
  return <h1>{t('title')}</h1>;
}
```

**Client components (interactive only):**

```ts
'use client';
import { useTranslations } from 'next-intl';

export function GuestbookForm() {
  const t = useTranslations('guestbook.form');
  return <input placeholder={t('namePlaceholder')} />;
}
```

**Rule:** Default to `getTranslations()` in server components. Use `useTranslations()` only in `'use client'` components that require interactivity.

## Testing Strategy

### 1. Key Parity Test (Vitest)

`src/__tests__/i18n/keyParity.test.ts`

Recursively compares key sets of `en.json` and `ko.json`. Fails if any key exists in one but not the other. Enforces that both files stay in sync whenever a new key is added.

### 2. Middleware Routing Tests (Vitest)

- `accept-language: ko-KR` → redirects to `/ko/*`
- `accept-language: fr-FR` → redirects to `/en/*` (default)
- `accept-language: en-US` → redirects to `/en/*`
- `/blog/some-post` with `accept-language: ko` → redirects to `/ko/blog/some-post`
- `/api/guestbook` → middleware skipped, no redirect
- `/_next/static/*` → middleware skipped
- `/fr/blog` → `notFound()` (unsupported locale, controlled 404)

### 3. E2E Test Updates (Playwright)

- Update all existing path references from `/blog` → `/en/blog`, etc.
- Add smoke tests for `/ko` routes (homepage, blog, guestbook)

## Migration Steps (high-level)

1. Install `next-intl`
2. Create `src/i18n/` config files and `messages/en.json` + `messages/ko.json`
3. Update `src/middleware.ts` to use `next-intl`'s `createMiddleware(routing)`
4. Move `src/app/` pages under `src/app/[locale]/`
5. Update `src/app/layout.tsx` to use `getLocale()` for `<html lang>`
6. Add `src/app/[locale]/layout.tsx` with `NextIntlClientProvider`
7. Add locale validation + `notFound()` in `src/i18n/request.ts`
8. Replace all inline TEXT constants with `getTranslations()` / `useTranslations()` calls
9. Localize metadata in each page using `metadata` sub-keys
10. Add key parity test
11. Update E2E test paths
